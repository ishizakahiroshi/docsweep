前回、Claude Code の常駐 context を 14% 削った [という記事](https://zenn.dev/ishizakahiroshi/articles/20260622-claude-code-context-diet) を書きました。`/context` を開いたら何もしてないのに 80k tokens 食われていて、CLAUDE.md の整理と settings.json の小細工で 69.5k まで落とした、という話です。地味な節約に見えて、結局のところ「ルールの整理整頓」が一番効きました。

その記事を書き終えて、しばらく経って CLAUDE.md をぼんやり眺めていたら、まだ削れる場所があったんです。

書きながら気付いた話なので、設計段階の生のまま記事にしておきます。実装後の Before/After は別途まとめる予定で、今回は「設計判断と最初の実装が終わって、これから検証する」スタンスです。

## 残ったままだった「1 文字略記」節

前回の整理では、CLAUDE.md の中身を「毎ターン必要なルール」と「トリガー時だけ参照すれば良いルール」に分けて、後者を別ファイルへ追い出しました。そのとき、残した「毎ターン必要」側にこんな節があったんです。

```text
## プロンプト先頭の `P` 略記（スクリーンショット参照）

ユーザーのメッセージが半角大文字 `P` で始まる場合...

## プロンプト先頭の `S` 略記（スキル起動）

ユーザーのメッセージが半角大文字 `S` で始まる場合...
```

`P` を打てば最新スクショを Read、`S` を打てば skill を起動。タイピング 1 文字で「最短で意図を伝える」自作ショートカットで、長く愛用していました。

ただ、節としては 50〜80 行ほどあります。これが毎セッション、CLAUDE.md と一緒に常駐ロードされていました。

書きながら本音を言うと、「毎ターン独立判定」を AI 側に強制しているこの規則、AI の判断負荷も実は重いはずなんです。`P` で始まる通常英文（`Python`、`Project`、`Pull`、`Please`...）と衝突しないよう、ガードルールも自分で書き足していました。「`P` の直後が英字なら通常テキスト扱い」みたいな例外規則も含めて、50〜80 行になっていた、というのが正確なところです。

## 「これも slash command 化できるのでは」

Claude Code の slash command は `.claude/commands/<name>.md` に置くだけで `/<name>` で起動できます。**呼ばれた時だけ md がロードされる**ので、CLAUDE.md と違って常駐コストがゼロ。

`/P` という slash を作ってしまえば:

- CLAUDE.md からは規則本文を追い出せる（行数ぶん常駐コストが減る）
- AI は「毎ターン入力先頭を判定する」必要がなくなる（誤発火もゼロになる）
- ロジック本体は別 md に集約できる

タイピング数だけ見ると `P` (1 打) → `/P` (2 打) で微増するんですが、Tab 補完が効くので体感は大差ないです。

これ、`P` `S` の 2 個だけだと「節約効果と書き換え手間が釣り合うかな?」という規模なんですが、ちょうど同じ頃に [docsweep](https://github.com/ishizakahiroshi/docsweep) という自作ツールを使い込んでいて、「`/D` を作って docsweep の triage を 1 文字で呼びたい」気持ちが芽生えていました。

なら `/P` `/S` `/D` の 3 兄弟で揃えて設計したほうが、共通仕様が固まって将来 `/N` や `/R` を増やすときも楽になります。一気にやろう、と決めました。

## `/P` と `/S` の設計

3 兄弟の共通形をこう決めました。

| 形式 | 動作 |
|---|---|
| `/X` | 引数なし → X 別の既定動作 |
| `/X <自然言語クエリ>` | クエリ解釈して内部コマンドへ翻訳・実行 |

slash の本体ロジックは「`$ARGUMENTS` を読んでこのルールで翻訳・実行せよ」という指示書として md に書くだけ。既存の skill や CLI 本体には手を触れません。slash 側は薄いラッパーに徹する責務分離です。

`/P` の中身（抜粋・整形）はこんな感じです。

```markdown:~/.claude/commands/P.md
---
description: 最新スクリーンショット N 枚を読む（screenshot-read skill ラッパー）
---

`$ARGUMENTS` を読んで以下のルールで `screenshot-read` skill を起動せよ。

1. 数字 + 「枚 / 件 / 個 / つ」→ その数字を N とする
2. 数字単独 → N とする
3. 全角数字・漢数字も同様
4. 残りの自然言語 → 焦点指示として skill に渡す
5. 数字が無ければ N=1（既定）
```

これで、こんな感じに自然言語でクエリが書けます。

```text
/P                       → 最新 1 枚
/P 3枚                   → 最新 3 枚
/P 3枚 右上のエラー見て  → N=3 + 「右上のエラー」を焦点に
/P このスクショの数字読み取って → N=1 + 「数字読み取って」
```

`/S` も同じ骨格で、`$ARGUMENTS` を `~/.claude/skills/SKILLS.md` の起動語と突き合わせて、合う skill を 1 つ特定して起動する形にしました。引数なしのときは SKILLS.md 全文をドンと出すと長すぎるので、「名前 / 起動語 / 短い説明」の 3 列だけスリムに整形して見せる、というルールも書いておきました。

## 副産物の `/D` を 3 兄弟で揃える

docsweep 用の `/D` も同じ流儀で作りました。クエリ意図に応じて `triage` / `sweep` / `promote` / `summary` / `scan` のサブコマンドに翻訳して実行します。

```text
/D                            → 全プロジェクトの残作業
/D このディレクトリの残課題は？ → cwd のリポを --project に渡す
/D many-ai-cli の残作業ある？  → --project many-ai-cli
/D many-* の triage           → グロブで複数プロジェクトを束ねる
```

ここで判断が要ったのが「破壊系をどう扱うか」です。docsweep には `sweep`（archive 移送）や `promote`（様子見を完了に昇格して移送）という、ファイルを実際に動かすサブコマンドがあります。

正直、最初は「文意で判断させればいい」と思っていました。「やって」と書かれていれば実行、「候補は？」なら dry-run、というやつです。

ただ少し考えて、やめました。archive 移送は不可逆操作で、自然言語の文意判定に任せると AI の解釈ブレで誤実行する事故が起きやすい。`/D` から実行コマンドが直接打てるようにしておくと、ある日「`/D` ここを整理して」と打ったら本当にファイルが消えた、みたいな未来が見えました。

なので `/D` は **読み取り＋下見専用** の口に固定して、`sweep` / `promote` は **クエリに「やって」と書かれても常時 `--dry-run`** にしました。実行は別途自分で `python -m docsweep sweep --project xxx` を打つ運用。1 ステップ増えますが、不可逆操作に 1 ステップの摩擦は安いものです。

## `P3` の数字直結を捨てる覚悟

3 兄弟設計でひとつ、捨てなきゃいけないものがありました。`P3` の **数字直結記法** です。

旧 `P` 略記では、`P3` と打てばスペースなしで「最新 3 枚」が読めました。slash 化すると `/P3` を別 slash として作るのは ugly すぎて、`/P 3` という形にスペース 1 つ挟まざるを得ません。

`P3` は体に染み付いていたので、最初は捨てるのが惜しかったんです。

でも、設計を詰めていくうちに気付いたんですが、`/P` 化したことで **逆に自然文として柔軟になっている** んですよね。

旧形式（略記 + 焦点指示の 2 部構成）:

```text
P3 右上のエラー見て
```

新形式（自然文 1 本）:

```text
/P 3枚 右上のエラー見て
/P 最新 3 つの右上見て
```

新形式のほうが、N と焦点指示を 1 文に溶かせる柔軟性があります。`P3` の打鍵速度を失う代わりに、自然文の表現力を取り返した、と言える気がします。記事を書きながら気付いたんですが、これはトレードオフというより、置き換えてみたら新しい UX のほうが良かった、という話です。

## 設計段階で迷ったこと

ここに、実装に入る前の生の迷いを置いておきます。半年後に「あのとき何を考えてたっけ」を思い出すためのメモです。

**数字抽出の表記揺れをどこまで対応するか。**

`3枚`、`3 枚`、`３枚`（全角）、`三枚`（漢数字）。半角数字 + 「枚」だけ拾えば 95% は捌けるはずですが、`/P 三つ` と打つ人（自分）もいる気がしました。結局、全角・漢数字までは対応するルールを md に書き込みました。「2 つ以上数字が含まれたら最初の `枚/件/個` 直前の数字を N とする」みたいなエッジケースルールも明記しています。やりすぎかなと思ったんですが、半年後に「`/P 三つ`」と打って動かなかったら間違いなく自分が混乱するので、入れておきました。

**「このディレクトリ」をどう解釈するか。**

`/D このディレクトリの残課題は？` の `このディレクトリ` は cwd を指すんですが、cwd の取り方が悩ましかった。`pwd` の basename を `--project` に渡すと、git リポのサブディレクトリにいるとき正しく解決されません。なので `git rev-parse --show-toplevel` で project root を取って、その basename を渡すルールにしました。git 配下じゃないときだけ `pwd` の basename にフォールバック。「ここ」「カレント」「いま居る場所」「current」も同じ意味で受けます。これは 3 兄弟共通仕様として `~/.claude/COMMANDS.md` に集約しました。

**複数プロジェクトをどう扱うか。**

最初は「単一 `--project` だけ対応、複数なら別々に打って」で済ませる気だったんですが、`/D many-* の triage` という書き方を 1 回試したくなって、グロブパターンを受けるようにしました。実装は CLI 本体を触らず、`/D` の md 本文ロジック内で「プロジェクト一覧取得 → パターンマッチ → 順次実行 → AI 側でマージ」する形に押し込めています。CLI の `--project` を複数値対応にすると後段フィルタの単純さが崩れるので、薄いラッパー側で吸収する判断です。

**過渡期の旧形式をどう扱うか。**

`/P` 化したあと、ユーザー（自分）がうっかり旧 `P3` を打ったときに AI がどうするか。「`/P` に移行済」と能動案内するか、自動翻訳して `/P 3` 相当を発火するか、何もしないか。

これも結構迷ったんですが、「何もしない」に倒しました。能動案内は毎ターン出るとノイズですし、自動翻訳すると「slash 版だけが正本」という単一ルールが崩れて挙動が二重化します。CLAUDE.md には「旧形式は発動しない」とだけ書いて、ユーザーが「あれ動かない」と気付いたタイミングで自分で `/P` に切り替える、というシンプルな運用にしました。

## 実装後の Before/After は次回

ここまで設計と最初の実装（3 つの slash command 作成、CLAUDE.md の旧略記節を移行注記 2 行に縮約）は終わっています。`~/.claude/commands/P.md` `S.md` `D.md` `README.md` が並び、CLAUDE.md からは 50〜80 行ぶんの常駐ルールが消えました。

ただ、これで本当に節約できているのか、3 兄弟の使い心地はどうなのか、という **実測** はこれからです。1 週間くらい使い倒してから、`/context` の Before/After 数値と運用感の続報を書く予定です。

連載感が出てきました。前回が「常駐コストを `/context` で可視化して、ルール棚卸しで 14% 削った」回。今回が「残ったルール本文を slash 化して、さらに削るための設計をした」回。次回が「1 週間使ってみたら本当に削れていたか」回、になる予定です。
